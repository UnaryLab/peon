# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

An always-on Slack multi-agent framework: one Python process runs several Slack bots, each its own Slack app, each backed by a CLI (`claude` or `codex`). Current agents (defined in `agents.json`):

- **Aristotle**: claude, `unarylab-research:research_manager`
- **Brunel**: claude, `unarylab-research:project_manager`
- **Cicero**: claude, general (no sub-agent)
- **Dijkstra**: codex, `project_manager` codex profile (from the unarylab-codex-marketplace, installed into `~/.codex`)

See `README.md` for installation and Slack usage, and `ARCHITECTURE.md` for the full internals (code layout, the backend abstraction, the verified CLI invocations, the session store, and hot-reload reconcile mechanics).

## Commands

Everything runs in the `peon` conda env (Python 3.12):

- Run all bots (always-on): `conda run -n peon python -m src`
- Full test suite: `conda run -n peon python -m pytest tests/ -q` (mocked/offline, no live Slack or CLI calls)
- Single test: `conda run -n peon python -m pytest tests/test_runner.py::<test_name> -q`
- No-pytest fallback runner: `conda run -n peon python tests/test_runner.py`
- Lint / format: `conda run -n peon ruff check src tests` / `ruff format src tests`
- Print a Slack app manifest: `python -m src manifest <name>` (add `--write` to dump `manifests/manifest-<name>.json`)

## Architecture (the parts that span files)

**Config is split on purpose.**
- `agents.json` is the single source of truth for agent *definitions* (`name`, `display_name`, `backend`, `model`/`effort`, plus `claude_agent` on claude entries only, and `codex_profile` on codex entries only). `codex_profile` is the optional codex-only persona analog of `claude_agent`: the NAME of an operator-installed `~/.codex/<name>.config.toml` profile (whose `developer_instructions` is the persona), which `codex_runner` applies as `--profile <name>` on the fresh run since `codex exec` has no agent flag (model/effort still come from `agents.json` via the CLI flags). `src/agents.py` loads it into `REGISTRY`.
- `.env` is *authoritative* runtime config (secrets, the `*_TIMEOUT_MIN` timeouts, `SESSIONS_PATH`, and the feature toggles below; NOT model/effort, which live only in `agents.json`). `src/env.py`'s `load_env()` runs with `override=True` as the FIRST executable code in `src/__main__.py`, before anything reads the environment. That ordering is why `.env` beats shell variables and why import-time reads pick it up.
- **Feature env vars** (all optional, all read LIVE per call so a SIGHUP `.env` reload toggles them; documented in `.env.example`): `SHOW_USAGE` (default ON: append the usage/cost/context-% footer), `STREAM_OUTPUT` (default ON: incremental streaming; `0` forces the byte-identical legacy argv), `WRITE_ALLOWLIST` (comma-separated Slack user/channel IDs allowed to enable write-mode; empty = fail-closed/deny all), `WORKDIR_BASE` (base dir for per-thread isolated write workdirs, default `~/Projects/.peon-workdirs`, an absolute path OUTSIDE the repo on purpose so a write-mode agent's edits never touch the framework source; set `WORKDIR_BASE` to override), `CONSENT_TTL_MIN` (write-mode consent lifetime in minutes, default 2880, i.e. 2 days).

**One process, one Slack app per agent.** `src/app.py` (a facade over `src/slack/`) builds one Bolt `App` + `SocketModeHandler` per *startable* agent (both `SLACK_BOT_TOKEN_<NAME>` and `SLACK_APP_TOKEN_<NAME>` present), each on its own thread. Agents missing tokens are skipped with a warning (partial startup). Each app dispatches to its own fixed agent; there is no keyword routing.

**Message handling is dedup'd and async.** A Slack event can arrive as BOTH an `app_mention` and a `message.*`, so the handler calls `seen_before(<event id>)` (an in-process bounded dedup, defined in `src/runners/common.py` and re-exported by the `claude_runner` facade) to answer it once. It acks Slack immediately, fetches a bounded visible Slack-thread transcript (`conversations.replies`, prior messages only) for context, posts a "thinking" placeholder, runs the slow CLI in a background thread, then `chat_update`s the placeholder with the reply, so Slack's ~3s ack deadline is never blocked. The transcript lets another agent read Slack-visible output in the same thread; it does not share hidden CLI session state.

**Backend abstraction (`src/runners/`).** `get_runner(backend)` returns the `claude_runner` or `codex_runner` FACADE module (the Claude/Codex internals live in `src/runners/claude.py` / `codex.py`; the shared dedup `seen_before` and the run-cancel `Interrupt` token live in `src/runners/common.py`; codex no longer imports claude). The unified seam is `runner.answer(agent, prompt, prior_session_id, overrides=None, on_update=None, cancel=None) -> (reply_text, session_id, meta)`, so `app.py` is backend-agnostic: `get_session -> runner.answer -> set_session`. `meta` is `{context_pct, tokens, cost_usd, duration_s}` (any field `None`); `overrides` is the per-thread model/effort/write dict; `on_update(partial_text)` drives streaming; `cancel` is an optional `Interrupt` the worker passes so a `!stop` can SIGINT the streaming run (see Run interrupt below).

**Per-thread JSON stores (vendor-neutral `src/store/`, re-exported by the `claude_runner` facade, all beside `sessions.json`).** Four stores share one lock (`_SESSIONS_LOCK`), one dir (`os.path.dirname(_sessions_path())` via `_sibling_store_path`), and `SESSIONS_PATH` redirects all of them at once. `store/base.py` is the single source of truth for the lock + path resolution; `sessions.py`/`overrides.py`/`crons.py`/`consent.py`/`workdir.py` are the per-store modules. When a store fn is called with `path=None`, `store.base._resolve_path` resolves the path through the LIVE `claude_runner.<resolver>` (so a test's `setattr(claude_runner, "_sessions_path", ...)` still redirects every store), falling back to the local resolver. Dict-shaped stores use `_load_dict_store`/`_save_dict_store`:
- `sessions.json`: `(agent_name, thread_ts) -> session_id`. Claude stores a `--session-id` UUID (resumed via `--resume`); Codex captures and resumes its own `thread_id`. Each (agent, Slack thread) is its own conversation; context survives a restart. Used by BOTH backends (driven by `app.py`); not Claude-specific.
- `overrides.json`: `(agent_name, thread_ts) -> {model?, effort?, write?, write_expires_at?}`, set by the `!model`/`!effort`/`!write`/`!reset` control phrases. `write`+`write_expires_at` back the TTL-bounded write-mode grant (`is_write_active` is the single read-time gate).
- `crons.json`: a LIST of `{id, schedule, agent, channel, thread_ts, prompt, enabled}` (its own `_load_crons`/`_save_crons` since it is list-shaped).

**Telemetry footer (`SHOW_USAGE`).** When `SHOW_USAGE` is truthy, `app._format_usage(meta)` appends a one-line `ﾂｷ N% ﾂｷ X tok ﾂｷ $Y ﾂｷ Zs` footer (each field dropped when `None`). Default OFF; read live so a SIGHUP `.env` reload toggles it.

**Streaming (`STREAM_OUTPUT`).** Default ON. Both runners read the CLI's JSONL stdout incrementally and call `on_update`; `app._make_stream_updater` throttles `chat_update` to ~1/sec, with an unconditional final update. `STREAM_OUTPUT=0` forces the legacy single-blob path whose argv is byte-identical to the pre-streaming code. Claude streaming argv adds `--output-format stream-json --include-partial-messages --verbose`; Codex argv is unchanged (it already emits `--json`).

**Control phrases (one dispatcher).** `app._handle_control_phrase` matches `CONTROL_RE` (`^!(model|effort|reset|write|cron)\b ...`) on the de-mentioned prompt and dispatches; a handled phrase acks into the thread and does NOT run the agent. `!model`/`!effort`/`!reset` mutate `overrides.json`; `!write on|off|status` toggles write-mode (ON gated by consent); `!cron add|list|remove|on|off` mutates `crons.json`. Interrupt phrases (`!stop`/bare `stop`/`ctrl-c`/`^c`/`interrupt`) are matched BEFORE the `!`-gate and signal the thread's in-flight run (see Run interrupt below).

**Run interrupt (`!stop`).** A `!stop` (or bare `stop`/`interrupt`/`ctrl-c`/`^c`) interrupts the run in flight for that `(agent, thread)` without starting a new one (matched BEFORE the `!`-gate, since bare forms carry no `!`). `src/runners/common.py`'s `Interrupt` token holds the live `Popen`; `.request()` sets a flag and SIGINTs it (mimics Ctrl-C, letting the CLI flush its session). `src/slack/interrupt.py` is the in-memory `{(agent, thread): Interrupt}` registry: the worker `register`s a token, passes it as `runner.answer(..., cancel=token)`, and `unregister`s in a `finally`. On the SIGINT-induced nonzero exit the streaming loop checks `cancel.requested` and RETURNS the partial reply instead of raising, so `set_session` still persists the id and the thread stays resumable (the worker marks it `_(interrupted)_`). STREAMING ONLY: under `STREAM_OUTPUT=0` there is no `Popen` handle, so `!stop` is a no-op.

**Write-mode SAFETY MODEL (read-only by default).** Runs are read-only unless a thread has an ACTIVE write grant. `!write on` is allowed only for a principal in `WRITE_ALLOWLIST` (fail-closed: empty allowlist denies everyone) and posts Block Kit Approve/Deny buttons; an allowlisted Approve calls `grant_write_consent` for `CONSENT_TTL_MIN` minutes. While active, the worker injects `_workdir` (an isolated dir under `WORKDIR_BASE`, namespaced by agent+thread; default `~/Projects/.peon-workdirs`, an absolute path OUTSIDE the repo on purpose so a write-mode agent's edits never touch the framework source) into `overrides`, and the runners relax the sandbox: claude adds `--permission-mode acceptEdits --add-dir <workdir>` and sets cwd; codex flips `-s`/`sandbox_mode` to `workspace-write` and sets cwd. On expiry the run reverts to the read-only (byte-identical) argv automatically. The consent buttons arrive over Socket Mode (no request URL) but require interactivity enabled in the Slack app manifest.

**Files in/out.** Inbound: `files[]` on a message are downloaded with the bot token (`_http_get_bytes`) into a per-thread temp dir and their paths appended to the prompt. Outbound: after a run, files created/modified in the thread's write workdir (mtime since run start) are uploaded back via `files_upload_v2`. Requires `files:read`/`files:write` scopes.

**Cron (Slack-native, in-process).** A daemon thread (`_scheduler_loop`) ticks every 60s, re-reads `crons.json`, and fires each enabled entry whose 5-field expression (`cron_matches`, a hand-rolled matcher, no dependency) matches the current minute, synthesizing a run through the same `_run_and_update` seam. This is distinct from Claude Code's own `/schedule` (cloud routines).

**Model/effort source:** each agent's `model` and `effort` come SOLELY from its `agents.json` entry (the single source of truth, set on every shipped entry). There is no env-var override. `agents.resolve` reads the field and, only if it is missing/empty, applies a single code-level fallback (logging a warning): claude model falls back to the pin `claude-opus-4-8[1m]`, everything else to omit. Claude always emits `--model`; effort and the Codex model are omitted when their field is absent.

**Manifests are generated** from `agents.json` by `src/manifest.py`; `manifests/` is gitignored, not a source of truth.

## Things that bite

- **Three module paths are FACADES; keep the re-exports complete.** `src/runners/claude_runner.py` (re-exports `src/runners/claude.py` + `common.{seen_before, Interrupt}` + the whole `src/store/*` surface), `src/runners/codex_runner.py` (re-exports `src/runners/codex.py`), and `src/app.py` (re-exports `src/slack/*`) exist BECAUSE the test suite references symbols and monkeypatches attributes on these EXACT paths, and `get_runner` returns these objects (`get_runner("claude") is claude_runner`). So: keep `subprocess` (runner facades) and `tempfile`/`runners` (app facade) exposed as facade attributes (Kind-A shared-singleton patch targets); the Kind-B names (`_run_and_update`, `_scheduler_tick`, `_fire_cron`, `_now`, `build_app_for`, `SocketModeHandler`, `reconcile`, `_attachments_dir`, `_http_get_bytes`) are resolved by their cross-module callers THROUGH the facade via a lazy in-body `from src import app as _appfacade`, and the store-path resolvers are resolved through `claude_runner` via `store.base._resolve_path`. When you add a symbol to a store/slack/runner submodule that a test references on one of these paths, re-export it from the relevant facade.
- **Adding an agent: reload or restart.** Flow: add an entry to `agents.json`, run `python -m src manifest <name>`, create the Slack app *From a manifest* (enable Socket Mode, get the token pair), add `SLACK_BOT_TOKEN_<NAME>`/`SLACK_APP_TOKEN_<NAME>` to `.env`, then either **hot-reload** (`kill -HUP <pid>`, or `systemctl reload`, which sends SIGHUP) or restart. In-flight conversations resume from `sessions.json` either way.
- **Hot-reload via SIGHUP (`src/app.py`, no restart).** `kill -HUP <pid>` (or `systemctl reload`) makes the running process re-read `agents.json` + `.env` and reconcile live connections by the **delta only**: added agents are connected, removed/now-unstartable ones are closed (`handler.close()`) and dropped, agents whose `agents.json` entry **or** resolved token changed are restarted, and **unchanged agents keep their live connection** (conversations are not interrupted). "Changed" is a per-agent snapshot of the entry + its two token values. Mechanism: the SIGHUP handler just sets a `threading.Event`; `main()`'s reload loop (which replaced the old `threading.Event().wait()` idle) does the reconcile on the main thread via `app.reconcile(live)`. **Crash-safe:** a missing/invalid `agents.json` or any reconcile error is logged and the running agents are left untouched (the process never dies). `agents.reload()` re-reads the registry and mutates `REGISTRY` in place. **Caveat:** finish editing BOTH `agents.json` and `.env` before sending the signal, since one SIGHUP reads both at once. POSIX only.
- **The built CLI invocations are load-bearing.** They were verified against the real CLIs; do not change the argv without re-verifying. Claude (read-only, non-stream = the asserted default path): `claude -p --output-format json {--session-id|--resume} <id> [--agent <x>] --model <m> [--effort <e>] <prompt>`. Codex (read-only): `codex exec [resume <tid>] --json --skip-git-repo-check {-s read-only | -c sandbox_mode=read-only} [--profile <p>] [-m <m>] [-c model_reasoning_effort="<e>"] -o <lastmsg_file> <prompt>` (Codex mints its own `thread_id`, which cannot be pre-set like Claude's session id; `--profile` is fresh-run only). **Feature-conditioned argv variants** (still byte-exact, asserted in lockstep): with `STREAM_OUTPUT` on (the run-time default), claude swaps `--output-format json` for `--output-format stream-json --include-partial-messages --verbose` (codex argv is unchanged, it already emits `--json`); with write-mode ACTIVE, claude adds `--permission-mode acceptEdits --add-dir <workdir>` and codex flips the sandbox value to `workspace-write`. The write-OFF and `STREAM_OUTPUT=0` paths must stay byte-identical to the pre-feature argv.
- **Tests are hermetic and offline.** A `conftest.py` autouse fixture pins `STREAM_OUTPUT="0"` (the legacy non-stream path) so the runner unit tests that mock `subprocess.run` assert the single-blob default-path argv deterministically; streaming tests opt back in with `monkeypatch.setenv("STREAM_OUTPUT", "1")`. Model/effort need no env scrub: they come SOLELY from `agents.json` (no env-var layer), so a developer's shell can no longer leak into the argv assertions. All subprocess and Slack I/O is mocked. `tests/test_runner.py` also carries a no-pytest fallback runner with a hand-maintained `tests = [...]` list (which sets the same `STREAM_OUTPUT="0"` default itself, since it does not load conftest); when you add a test function, add it to that list too.
- **Behavior-preserving changes must keep the default-path argv byte-identical** (the suite asserts exact argv). `.env` is never edited by tooling: change `.env.example` and have the user mirror it into `.env`.
- **Slack app reconfig for files + consent.** The file in/out feature needs the `files:read` and `files:write` bot scopes, and the write-mode consent buttons need **interactivity** enabled in each Slack app (the action payloads arrive over Socket Mode, so no request URL is needed, but interactivity must be on). The shipped manifest (`src/manifest.py`) carries these; after pulling these features, recreate/update each Slack app from its manifest (or add the scopes + interactivity by hand) and reinstall so the new scopes take effect.
